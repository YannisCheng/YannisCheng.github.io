Mac环境下终端中配置Groovy

Groovy是依赖于Java的，所以首先要配置好JDK。

下载地址：http://groovy-lang.org/download.html 下载后，进行解压缩。

配置Groovy环境变量

vim ~/.bash_profile

输入

# groovy
export PATH=$PATH:/Users/xuhongchuan/install/groovy-2.4.4/bin

保存后让配置生效并打印版本号

source ~/.bash_profile
groovy -v

显示结果为：

Groovy Version: 2.4.4 JVM: 1.8.0_45 Vendor: Oracle Corporation OS: Mac OS X

创建一个groovy文件

vim hello.groovy

输入

println "Hello Groovy"

保存后用groovy命令运行

groovy hello.groovy

输出结果为：

Hello Groovy

Gradle基本：

Gradle是一个能通过插件形式自定义构建逻辑的优秀构架工具。和Maven一样，Gradle只是提供了构建项目的一个框架，真正起作用的是Plugin。
Gradle本身只是一个架子，真正起作用的是Task和Plugin。

  • 领域驱动设计：Gradle本身的领域对象主要有Project和Task
  • 基于Groovy脚本语言构建
  • 在gradle中编写代码以控制编译过程
  • DSL：领域专用语言 - 描述和控制构建逻辑
  • Task：Gradle属于任务驱动型构建工具。而构建过程基于Task

DSL（Domain Specific Language）：领域特定语言

释义

其基本思想是“求专不求全”，不像通用目的语言那样目标范围涵盖一切软件问题，而是专门针对某一特定问题的计算机语言。

简洁

  • 从概念上说，程序的编写过程就是把业务领域中的问题通过代码或者程序模型表达出来，计算机的程序模型较为单一（归根结底都是运算和存储)。
  • DSL的简洁性往往是一种思维上的简洁性，使我们不用费太多的气力就能看懂代码所对应的业务含义。

领域驱动设计

Project

Project为Task提供了执行上下文，所有的Plugin要么向Project中添加用于配置的Property，要么向Project中添加不同的Task。

Task

一个Task表示一个逻辑上较为独立的执行过程，比如编译Java源代码，拷贝文件，打包Jar文件，甚至可以是执行一个系统命令或者调用Ant。另外，一个Task可以读取和设置Project的Property以完成特定的操作。

Gradle文件说明：

  • build.gradle (Project，全局公共的配置)
  • build.gradle (Module，控制每一个module具体配置).
  • local.properties (SDK Location.，只做了一件事：指定了一个SDK路径)
  • gradle.properties（project Properties， 使用Gradle脚本中的动态参数）
  • settings.gradle (Project Settings，配置Gradle的多项目管理)
  • gradle-wrapper.properties (Gralde Version)

Gradle 文件内容解释

  • buildscript { } ：配置了用于构建整个project的代码。
  • apply plugin: 'com.android.application' ：该插件表示编译的是Android应用。
  • android { } ：配置了Android构建所需的参数，是Android DSL的入口。
  • dependencies { } ：为单个Module添加依赖

Gradle的项目结构

Gradle文件结构

有两个最为基本的组件source sets，分别存放在：

- src/main/        ：    存放应用代码
- src/androidTest/        ：    存放测试代码

其中，Android Plugin特有的，在src/main/目录下： 

- java/         ：    存放`*java`文件的路径为：src/main/java
- res/
- assets/
- aidl
- rs/
- jni/
- jniLibs/
- AndroidManifest.xml        ：    自动创建

自定义项目结构

Task 任务

  • task任务之间有依赖

  • 支持驼峰命名法的task简称

    gradle aR

    等价于：

    gradle assembleRelease

通用Task

添加一个plugin后，就会自动执行一些列的构建任务（build tasks）。
java plugin和android plugin都会执行的task：

  • assemble ：组合项目所有的输出任务
  • check ：执行所有的检查任务
  • build ：执行 assemble和check两个task的所有工作

注意：以上3个task不做任何事情，只是一个task标志，用来告诉plugin添加实际需要执行的task来完成工作，即：task集

  • clean ：清空项目的输出

Android Plugin 增加了新的标志性的Task

  • connectedCheck ：在一个连接的设备或者模拟器上执行检查，他们可以在所有连接的设备上并行执行检查
  • deviceCheck ：通过APIs连接远程设备来执行检查，主要用于CI（持续集成）服务生

一个Android项目至少有2个输出：

  • assemble
    • assembleDebug ：debug APK
    • assembleRelease ：release APK

获取更多Task

命令环境中，获取更多task：

gradle tasks

查看所有task列表和他们之间的依赖关系：

gradle tasks --all

动态的构建文件 - Mainfest属性

可以从一个文件中或者其他自定义的逻辑代码中读取信息。
eg：

def computeVersionName(){
    }

android {
    compileSdkVersion 26

     defaultConfig {
         applicationId "com.yannischeng.temp"
         minSdkVersion 26
         targetSdkVersion 26
         versionCode 1 

         //versionName "1.0"，使用自定义方法动态获取数据，
         //切记：自定义方法时要避免与系统自带的方法重名。
         versionName computeVersionName()
    }  
 }

构建类型：生成不同的apk类型 - buildTypes

Android Plugin 默认构建 debug和release两个版本，区别为：能否在安全设备上调试及如何签名。
debug使用过程中通过name/password对生成秘钥进行签名，release构建过程中无需签名，需要自行签名。

签名配置 - signingConfigs

签名文件需要：

  • keystore 文件
  • keystore 密码
  • key alias
  • key 密码
  • 签名文件类型

signingConfigs由 签名文件路径 + 类型 + key名 + 密码组成。
debug使用debug keystore（默认的storePassword、keyAlias、keyPassword），其路径为：￥HOME/.android.debug.keystore。

配置例子：

android {
    signingConfigs {
        debug {
            storeFile file("debug.keystore")
        }
        customConfig {
            storeFile file ("other.keystore")            storePassword "android"
            keyAlias "aliaskey"
            keyPassword "android"
        }
    }
    buildTypes {
        foo {
            signingConfig signingConfigs.customConfig
        }
    }
}

依赖、Library、多项目

包依赖

本地包依赖

dependencies {
    compile fileTree(dir: 'libs', include: ['*.jar'])    }

android {
}

compile配置将被用于编译 main application。其中所有的依赖会被添加到编译 classpath中，同时也会被打包到最终的apk中。

  • compile ：编译主module
  • androidTestCompile ：编译主module的测试
  • debugCompile ：debug Build Type的编译
  • releaseCompile ：release Build Type的编译

远程包依赖

Gradle支持从Maven或者Lvy仓库中拉取依赖文件。首先将仓库添加到列表中，然后声明。

repositories {
     //jcenter()是指定仓库URL的便捷方法。
     //Gradle会遵循依赖关系的传递性，若是一个库依赖于其他库，则被依赖的库也会被拉取
        jcenter()
    }

dependencies {
       compile ' ' 
}  

android {
}

多项目设置

在项目根目录下有一个settin.gradle文件用于声明所有项目。该文件定义了各个gradle项目的位置：

include ':echarts_demo', ':resourcedemo', ':mediarecorderdemo', ':hotfix', ':temp'

Library项目

创建Library项目

Library项目和常规的Android项目只有小部分差异。两个插件内部共享大部分代码。

buildscript {
    repositories {
        jcenter()
    }
    dependencies {
    }
}

//表示这是一个库
apply plugin: 'com.android.library'

android{
}

一个Library项目例子：

apply plugin: ‘com.android.library’

android {
    compileSdkVersion 22
    buildToolsVersion '25.0.0'

    defaultConfig {
        minSdkVersion 11
        targetSdkVersion 22
    }

    buildTypes {
        release {
            minifyEnabled false
            proguardFiles getDefaultProguardFile('proguard-android.txt'), 'proguard-rules.txt'
        }
        debug {
            minifyEnabled false
            proguardFiles getDefaultProguardFile('proguard-android.txt'), 'proguard-rules.txt'
        }
    }

    sourceSets {
        main {
            manifest.srcFile 'AndroidManifest.xml'
            java.srcDirs = ['src']
            resources.srcDirs = ['src']
            aidl.srcDirs = ['src']
            renderscript.srcDirs = ['src']
            res.srcDirs = ['res']
            assets.srcDirs = ['assets']
            jniLibs.srcDirs = ['libs']
        }

        instrumentTest.setRoot('tests')
        debug.setRoot('build-types/debug')
        release.setRoot('build-types/release')
    }
}

dependencies {
    compile fileTree(dir: 'libs', include: ['*.jar'])
}

普通项目和Library项目的区别

  • LIbrary项目会输出.aar包（标准的Android 包），其中包含编译文件(jar或者.so文件)和资源文件(manifest，res，assets)。
  • Library项目同样可以借助普通项目生成测试apk测试，标识task也相同，所以在命令行的构建方式也是一样的。
  • 大部分BuildType的配置不适用与Library项目。但是可以自定义sourceSet

引用Library项目

dependencies {
        compile project(':KSYMediaPlayer')
        compile project(':KSYMediaWidget')
}

注意：引用多个Library时，引用的先后顺序十分重要。

Library项目发布

默认LIbrary只会发布release variant版本。

android {
    defaultPublishConfig "debug"
}

若使用其他 flavor（特性） 的variant版本，则：

android {
    defaultPublishConfig "flavorDebug"
}

发布所有variant版本：

android {
    publishNoDefault true
}

Glide有 默认文件 概念，使用如下：

compile project(':libraries:lib2')

添加其他依赖，则需要指定具体使用哪一个：

dependencies {
        flavor1Compile project(path: ':lib1', configuration: 'flavor1Release')
        flavor2Compile project(path: ':lib1', configuration: 'flavor2Release')
}

注意：已发布的configuration是一个完整的variant版本。
